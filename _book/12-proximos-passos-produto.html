13.1 Objetivo deste capítulo

Este capítulo consolida o Quarto Book como uma entrega metodológica e executável. O pipeline desenvolvido demonstra como combinar CID, regras textuais e LLM para classificar atendimentos em síndromes febris, mantendo rastreabilidade das decisões e condições explícitas de auditoria.

O material foi construído com base sintética, pois a base real ainda não foi incorporada. Mesmo assim, a arquitetura já está organizada para ser adaptada para dados reais, execução operacional e futura transformação em aplicativo Shiny.

  3. Arquitetura flexível de LLM, permitindo uso em modo mock, Ollama local, Gemini ou API OpenAI-compatible.

13.4 Como adaptar para a base real

A primeira etapa de adaptação para dados reais é substituir a base sintética por uma tabela ou arquivo com os campos mínimos necessários.

Ver código

  caption = "Campos mínimos recomendados para adaptação à base real"
)
Campos mínimos recomendados para adaptação à base real
campo obrigatoriedade observacao
record_id obrigatório Identificador único do atendimento.
data_atendimento obrigatório Data ou data/hora do atendimento; usada nas séries históricas.
unidade recomendado Nome ou código da unidade de atendimento.

anamnese recomendado Texto clínico, anamnese, evolução ou campo livre equivalente.
texto_clinico obrigatório após padronização Campo consolidado usado pelas camadas regex e LLM. Pode ser criado a partir de queixa + anamnese.

Na prática, o campo mais importante para as camadas textuais é texto_clinico. Ele pode ser criado pela combinação de campos como queixa principal, anamnese, evolução, observação clínica e motivo do atendimento.

Exemplo conceitual:

texto_clinico = queixa + anamnese + evolução + observação

O pipeline não exige que a base real tenha exatamente os mesmos nomes de coluna da base sintética. O ponto central é que a etapa de normalização converta a entrada para o schema interno usado pelo projeto.

13.5 Configuração da camada LLM

A camada LLM foi desenhada para funcionar em modo mock, Ollama local, Gemini e APIs OpenAI-compatible. Para fins de reprodutibilidade da entrega, o Quarto Book renderiza por padrão em modo mock. Testes locais com Ollama podem ser executados alterando LLM_PROVIDER=ollama e LLM_MODEL conforme o modelo disponível na máquina.

LLM_PROVIDER=mock
LLM_MODEL=mock-sindromico-v1

Para uso local com Ollama:

LLM_PROVIDER=ollama
LLM_MODEL=mistral:latest
LLM_BASE_URL=http://localhost:11434

Para uso com Gemini:

LLM_PROVIDER=gemini
LLM_MODEL=gemini-1.5-flash
LLM_API_KEY=sua_chave

Para uso com API OpenAI-compatible:

LLM_PROVIDER=openai_compatible
LLM_MODEL=gpt-4o-mini
LLM_API_KEY=sua_chave
LLM_BASE_URL=https://api.openai.com/v1

A escolha do provedor deve considerar custo, infraestrutura, privacidade, conectividade, tempo de execução e política institucional de tratamento de dados sensíveis.

13.6 Validação antes do uso operacional

Antes de usar o pipeline em rotina real, recomenda-se uma etapa formal de validação com especialistas.

O processo mínimo recomendado é:

1. Selecionar amostra estratificada de atendimentos reais.
2. Rotular manualmente a síndrome principal por especialistas.
3. Comparar CID, regex, LLM e classificação final contra o padrão de referência.
4. Medir sensibilidade, especificidade, valor preditivo positivo e F1 por síndrome.
5. Revisar erros sistemáticos de classificação.
6. Ajustar dicionários, prompts, regras de prioridade e tratamento de CIDs genéricos.
7. Repetir a avaliação antes da implantação operacional.

A validação deve incluir casos simples, casos ambíguos, CIDs genéricos, textos incompletos, negação de febre e registros com múltiplos sintomas.

13.7.2 Telas mínimas recomendadas

A primeira versão do Shiny pode ter seis telas:

1. Visão geral
2. Importação e qualidade da base
3. Execução do pipeline
4. Resultados temporais
5. Auditoria de registros
6. Configurações

13.7.3 Tela 1 — Visão geral

Deve apresentar cartões com:

total de atendimentos
atendimentos classificados
percentual não classificado
número de síndromes detectadas
registros com revisão manual recomendada
última execução

A fonte principal pode ser tb_classificacao_final, combinada com tb_execucoes.

13.7.4 Tela 2 — Importação e qualidade da base

Deve permitir:

upload de CSV/Parquet
pré-visualização da base
mapeamento de colunas para o schema interno
validação de campos mínimos
percentual de textos vazios
percentual de CID ausente
distribuição temporal inicial

Nesta tela, o Shiny deve chamar as funções de normalização já existentes no projeto.

13.7.5 Tela 3 — Execução do pipeline

Deve permitir executar as camadas:

CID
regex
LLM
comparação
classificação final

A execução pode ser sequencial, com mensagens de progresso. Para bases grandes, recomenda-se processamento em lotes e gravação incremental no DuckDB.

13.7.6 Tela 4 — Resultados temporais

Deve reutilizar a lógica do capítulo de séries históricas:

série diária
série semanal
série por síndrome
ranking de semanas
ranking de unidades, se disponível

Os gráficos devem ser filtráveis por período, síndrome, unidade e faixa etária.

13.7.7 Tela 5 — Auditoria de registros

Essa é uma das telas mais importantes. Deve permitir revisar:

casos com revisão manual recomendada
casos com CID genérico refinado pelo texto
divergências CID × regex × LLM
regex e LLM divergentes
casos não classificados
justificativa da LLM
texto clínico original

A auditoria deve permitir exportar uma amostra para revisão por especialistas.

13.7.8 Tela 6 — Configurações

Deve permitir configurar:

provedor LLM
modelo LLM
URL base
modo mock ou real
número máximo de registros para teste
parâmetros de execução
limites de lote

Credenciais e chaves de API não devem ser gravadas no código. Devem ficar em .Renviron, variáveis de ambiente, cofre institucional ou mecanismo equivalente.

13.8 Cuidados para implantação operacional

Para uso em ambiente real, recomenda-se observar:

segurança e privacidade dos dados clínicos
controle de acesso
registro de logs de execução
versionamento de dicionários e prompts
separação entre ambiente de teste e produção
monitoramento de falhas da LLM
rotina de auditoria periódica
mecanismo de reprocessamento
backup do DuckDB ou migração para banco institucional

Se os dados clínicos forem sensíveis, o uso de LLM local pode ser preferível. Se uma API externa for usada, é necessário avaliar contrato, política de privacidade, anonimização e autorização institucional.

13.9 GitHub e versionamento

Para publicar o projeto em um repositório GitHub, recomenda-se incluir:

README.md
_quarto.yml
arquivos .qmd
pasta R/
pasta assets/
base sintética em data/
.Renviron.example
.gitignore
run_pipeline.R

Não se deve publicar:

chaves de API
base real com dados sensíveis
arquivos .Renviron
logs contendo informações pessoais
outputs de produção com dados identificáveis

A base sintética pode ser versionada, pois permite demonstrar o funcionamento do pipeline sem expor dados reais.

13.11 Fechamento

O produto atual deve ser entendido como um protótipo metodológico executável, pronto para demonstração e adaptação. Ele já contém as decisões centrais do pipeline, a lógica de persistência, a camada LLM desacoplada, a comparação entre métodos e a classificação final auditável.

A transição para uso operacional depende principalmente de três etapas: incorporação da base real, validação com especialistas e definição do ambiente de execução. A partir disso, o mesmo núcleo do pipeline pode ser reaproveitado em rotina automatizada, painel Shiny ou sistema de alerta.
